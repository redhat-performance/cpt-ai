### Performance Patterns

The synthetic data includes five types of performance patterns with realistic distributions:

1. **Stable** (60%): Performance varies within ±3%
   - Represents normal test variance
   - Most common in production environments
   - Small run-to-run variation

2. **Minor Improvement** (15%): Performance increases 5-12%
   - Incremental OS optimizations
   - Hardware tuning gains
   - Compiler improvements

3. **Improvement** (5%): Performance increases 15-35%
   - Major OS or hardware improvements
   - Significant optimization gains
   - Architecture enhancements

4. **Minor Regression** (15%): Performance decreases 5-12%
   - Minor performance impacts
   - Configuration changes
   - Usually still passes thresholds

5. **Regression** (5%): Performance decreases 20-45%
   - Significant performance regressions
   - Often marked as FAIL status
   - Critical issues requiring investigation

### Enhanced Features

#### Temporal Trends
- **Linear trends**: Gradual improvement or degradation over time
- **Seasonal patterns**: Cyclical variation simulating load patterns
- **Stable baselines**: Consistent performance across time periods

#### Metric Correlations
Realistic co-variance between related metrics:
- CoreMark: multicore and singlecore scores (r=0.85)
- CoreMark-PRO: CPU metrics correlate with summary scores (r=0.88-0.92)
- STREAM: Memory bandwidth operations correlate (r=0.90-0.95)
- FIO: Bandwidth and IOPS highly correlated (r=0.98)

#### Hardware-Specific Performance
Performance multipliers based on instance tier:
- **High-tier** (96 cores, 24xlarge): 115-125% of baseline
- **Medium-tier** (48-64 cores, 12xlarge): 95-105% of baseline
- **Low-tier** (16-32 cores, 4-8xlarge): 75-85% of baseline

#### Realistic Failures (~7.8% of tests)
- **Timeout** (2%): Test exceeds maximum execution time
- **Crash** (1%): Process termination with signals
- **Validation** (1.5%): Result checksum mismatches
- **OOM** (0.5%): Out-of-memory conditions
- **Unknown** (remaining): Miscellaneous failures

### Statistical Properties

Each metric includes statistical aggregations:
- `_mean`: Average value across test runs
- `_min`: Minimum observed value (≈ 96-99% of mean)
- `_max`: Maximum observed value (≈ 101-104% of mean)
- `_stddev`: Standard deviation (≈ 0.8-1.5% of mean)

These ranges provide realistic run-to-run variation while maintaining consistency.

## Regenerating Data

To regenerate the synthetic data with default robust parameters:

```bash
# Activate virtual environment
source venv/bin/activate

# Generate robust dataset (100 scenarios × 8 iterations = 800 documents)
python src/synthetic_data.py
```

### Custom Generation

For custom datasets with different parameters:

```python
from src.synthetic_data import SyntheticDataGenerator

generator = SyntheticDataGenerator(seed=42)

# Generate with custom parameters
documents = generator.generate_dataset(
    num_scenarios=150,              # More unique configurations
    iterations_per_scenario=10,      # More runs per config
    include_temporal_trends=True,    # Add time-based patterns
    include_failures=True            # Include realistic failures
)

generator.save_to_file(documents, "data/synthetic/custom_data.json")
```

### Parameter Guide

- **num_scenarios**: Number of unique test configurations (default: 100)
  - Each scenario = unique combination of OS version, cloud provider, instance type, and test type
  - Higher values = more diverse configurations
  - Recommended range: 50-200

- **iterations_per_scenario**: Test runs per scenario (default: 8)
  - Simulates repeated testing over time
  - Higher values = better statistical coverage
  - Recommended range: 5-15

- **include_temporal_trends**: Enable time-based patterns (default: True)
  - Adds realistic performance drift over time
  - Includes linear trends and seasonal variation
  - Useful for testing regression detection

- **include_failures**: Enable realistic test failures (default: True)
  - ~7-8% failure rate across all tests
  - Multiple failure types (timeout, crash, validation, OOM)
  - Essential for testing error handling

### Seed Control

The generator uses a random seed for reproducibility:

```python
# Reproducible dataset
generator = SyntheticDataGenerator(seed=42)

# Different dataset each run
generator = SyntheticDataGenerator(seed=None)
```

## Limitations

Current simplifications (compared to real production data):

- CPU flags are simplified (not exhaustive per model)
- Test-specific configuration parameters are minimal
- Failure logs are brief error messages (not full stack traces)
- NUMA topology is simplified (though scaled with core count)
- Network topology effects are not simulated in detail
- Multi-node/cluster configurations are not yet included
- Geographic latency variations are not modeled
